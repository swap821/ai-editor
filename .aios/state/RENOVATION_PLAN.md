## 1. Honest state + the renovation thesis

This is a real, ~2-week-plus supervised local-first AI-OS whose **backend core genuinely exceeds its blueprint** (security spine, hash-chained audit, scope-lock, approval capabilities, verifier, self-apply, multi-store memory, stigmergic learning, a cross-provider router — all end-to-end wired and proven by contract-level tests, 551 passed / 1 skipped) and whose **frontend has a genuinely production-grade polished CORE** (the 2D superbrain HUD: honest-dormancy, single-accent, spring physics, reactive reduced-motion, focus-on-approval). It is neither an MVP toy nor flawless. The renovation thesis is that **the work that matters is no longer "fix what's broken in the proven core" — it is the SEAMS the discipline never reached**: (a) a small set of one-config-change-away security/correctness hazards at the perimeter and in the legacy tier; (b) the marquee Jarvis VOICE loop, which is the operator's active priority and is mostly *wiring on existing seams* (the backend mind ships, a Web Speech STT loop sits half-built and orphaned in `App.jsx`); (c) the lower-tier surfaces beside the canon HUD — the workbench layer, the classic IDE's a11y/motion, a real responsive/touch story, observability/structured logging, and the test-darkness over the entire product face. None of the high-value work requires redesigning the cherished 3D brain. The honest target is **~90%, reported as such** — pursue completeness, never claim "100%".

**One honest deviation, stated up front (not buried):** `FUTURE_FRONTIER` ranks **VOICE *last*** of the three gaps (after the brain-ceiling and isolation), with explicit queue discipline ("do not let the frontier jump ahead of a better brain + default-strong isolation"). **This plan prioritizes the Jarvis VOICE mind FIRST this session** — Slice 1 (`POST /api/v1/chat`) has already shipped, and the operator has named the voice loop the active initiative. The operator's current priority wins; the deviation is deliberate and named. The frontier's reasoning still holds for the *deeper* voice work (local STT/TTS is a real RAM/latency bet on a 16GB box sharing GPU with Ollama) — so this plan splits voice into a cheap browser-native MVP (do now) and a deferred local-first hardening pass (respects the frontier's caution).

**Invariants honored.** All 6 `FUTURE_FRONTIER` invariants govern every item below — with the one operator update: **Invariant #6 (FIDELITY / "superbrain UI never redesigned") was RELAXED on 2026-06-14.** The 3D brain + space are now **MODIFIABLE but CHERISHED — enhance, never replace.** FIDELITY-IS-SACRED still governs *how* (no auto-degrade ever, canon tag + goldens before any visual change, before/after screenshots in HIS browser, reversibility via branch/tags). The other five invariants are untouched: evidence-over-model is absolute; RED is ALWAYS refused even after approval; human authority is never removed (only the earned, revocable, RED-un-earnable GREEN surface widens); local-first / privacy by default; the audit chain + frozen security core are sacrosanct.

## 3. Recommended execution order (first ~8 items, in sequence)

1. **P1-1 — Branch hygiene + push (S).** Do this *first*, before anything else: the entire real system has no off-box copy. The fast-forward chain is verified linear, so it's a 1–2 hour zero-risk move that also unblocks CI. (`PLAN.md` H0.)
2. **P0-1 — CORS guard (S).** The single highest-leverage perimeter fix and a clean win — a startup assertion + one regression test. Pairs naturally with the doc sweep that follows. (HIDDEN_KNOWLEDGE P0 #1.)
3. **P0-6 — App entrypoint (S).** Tiny, and it's a *prerequisite* for the proxy-header auth fix (P0-4), the observability/compose wrapper (P1-5), and a real deploy. Makes `AIOS_API_HOST/PORT` real.
4. **P0-2 + P0-5 — `reset_audit_chain.py` + legacy quarantine (S+M).** Do them together: quarantining the legacy tier is where the audit-reset script's repoint/banner belongs, and it removes the naming collisions that make the no-op silent. Pure-safety dead-code move with no behavior risk. (`PLAN.md` H3.)
5. **P1-10 — Doc-currency sweep (S).** Now that the baseline is clean (branches merged, CORS guarded, legacy quarantined), pin the ONE test baseline and reconcile the baseline-doc contradictions so the next builder isn't misled. Honor the doc-currency convention (superseded banners, never rewrite dated evidence). (`PLAN.md` H1.)
6. **P0-3 + P1-3 — Approval single-source-of-truth + session-id extraction (M+S).** P0-3 closes the worst UX failure (a strandable approval gate); P1-3 is a hard prerequisite for voice (the session-id fork trap). Sequence them back-to-back because **both are prerequisites for the voice loop**.
7. **P0-7 — Prompt input-shield (M).** Land the size-cap + injection-shield + rate-limit on `/api/v1/chat` *before* wiring voice traffic to it — voice will drive a new, easy-to-flood input path straight at a cloud provider with the operator's facts attached. Do it now so the voice loop is safe by construction.
8. **P1-2 (with P1-4 alongside) — Jarvis VOICE Slice 2 browser MVP + structured logging (M+M).** The marquee deliverable — wire the orphaned Web Speech STT to `/api/v1/chat` with TTS speak-back, the `voice-speaking` pulse, push-to-talk, the shared session-id, and full error-path UX. Bring up **P1-4 (structured logging)** in parallel so the new voice turn path is debuggable from day one and the audit-tamper alarm becomes durable. **Voice is a directive channel only — a spoken "yes" never redeems the approval token; the gateway re-classifies every transcribed token.**

**Rationale for the ordering:** the P0 security/correctness guards and the dead-code cleanup are deliberately front-loaded as cheap, low-risk wins that also *unblock* the higher-value work (entrypoint → deploy/observability; legacy quarantine → audit-reset fix; approval + session-id → voice; input-shield → safe voice). The Jarvis voice Slice 2 — the operator's named priority and the documented honest deviation from FUTURE_FRONTIER's queue — lands at step 8, immediately after its prerequisites are green, rather than first, because shipping it on an unguarded prompt surface with a fork-prone session id would be the wrong kind of fast. After these 8, the next wave is P1-9 (CI/coverage/typecheck, now that branches are merged), P1-6 (knowledge-graph traversal + recall-into-forge), P1-5 (observability/compose), then P1-7/P1-8 (workbench + classic a11y) and the P2 debt tier, with the P3 micro-detail workstream (P3-2) running as the operator's ongoing standing mandate on a canon-tagged, goldens-protected branch.
